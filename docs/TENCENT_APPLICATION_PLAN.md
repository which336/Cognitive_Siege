# 游戏策划投递执行计划

目标：把《认知围城》包装成一份能投递腾讯游戏策划的作品集材料，重点证明三件事：

1. 能把玩法概念落成可玩的原型。
2. 能用策划语言解释系统、数值、关卡和用户体验。
3. 能用数据和复盘证明自己会迭代，而不是只会堆创意。

## 一、投递定位

优先投递方向：

| 方向 | 匹配度 | 主打卖点 |
| ---- | ------ | -------- |
| AI 探索 / 创新玩法策划 | 高 | LLM 复盘真实改写波次、Boss 谈判影响战斗参数、fallback 保证演示稳定 |
| 系统策划 / 玩法策划 | 中高 | 塔、心魔、路线、塑形、经济障碍物、复盘演化形成完整循环 |
| 关卡策划 | 中 | 10 波节奏、Boss 节点、多路线压力，但还需要更强关卡调优记录 |
| 数值策划 | 中 | 当前已有 15 张 CSV 配置表、加载校验和配置报告 UI，但仍缺少成体系调参数据 |
| 文案 / 世界观策划 | 辅助 | 题材和人格化表达有亮点，但这不是本项目最强证明点 |

简历一句话定位：

> 独立设计并实现 AI 驱动自适应塔防原型《认知围城》：用 LLM 扮演战败心魔集体意识，将战斗数据转化为下一波真实策略改写，覆盖路线、编队、技能、阵容、Boss 谈判与 fallback 演示闭环。

## 二、必须先补齐的作品证据

### Step 1：稳定可试玩版本

目标：让面试官打开链接后 10-15 分钟内能看懂核心玩法。

我可以继续帮你做：

- 检查首屏、教程、按钮文案和 CODEX 是否足够清楚。
- 做一次完整试玩，记录卡点和演示翻车点。
- 修复明显的 UI、节奏、数值和可读性问题。

你需要做：

- 自己完整玩 3 局，记录每局是否通关、用时、失败波次、最难理解的点。
- 找 2-3 个同学或朋友试玩，要求他们边玩边说出自己看懂了什么。
- 不要解释玩法，让游戏自己解释；只记录他们卡在哪里。

验收标准：

- 第一次打开后 1 分钟内知道“放塔、开始波次、心魔到核心扣 SAN”。
- 第 5 波前能理解至少 3 种塔和 3 种心魔。
- 第 6 波开始能看懂“复盘影响下一波”。
- 单局时长控制在 12-18 分钟。

### Step 2：部署链接和演示视频

目标：简历里必须有可点开的作品入口。

我可以继续帮你做：

- 检查 GitHub Pages / Vercel / Netlify 部署配置。
- 写 README 顶部的作品入口和演示说明。
- 帮你整理 3-5 分钟演示视频脚本。

你需要做：

- 注册或准备 GitHub 仓库。
- 录制 3-5 分钟演示视频，最好包含你的口播。
- 视频必须出现：开局建塔、普通波、复盘面板、下一波变化、Boss 谈判、多路线压力。

演示视频结构：

1. 15 秒：一句话介绍项目和目标岗位。
2. 40 秒：核心循环，放塔、出怪、SAN、念力。
3. 60 秒：AI 复盘如何把战斗结果改成下一波策略。
4. 60 秒：多路线、心魔类型、Boss 谈判。
5. 45 秒：你做过的迭代取舍和下一步调优计划。

### Step 3：数值与关卡调优记录

目标：证明你是策划，不只是做了一个会动的 demo。

当前完成：

- 已加本地战斗统计导出功能。
- 终局面板会输出每波：通关/失败、SAN、漏怪数、击杀塔种、路线使用占比。
- 已建立 `docs/BALANCE_LOG.md` 调参记录模板。
- 导出 CSV 文件可作为后续 Run 记录和 Excel 调参证据。

你需要做：

- 至少完整记录 5 次调参。
- 每次调参写清楚：问题、假设、改动、结果、下一步。
- 不要只写“感觉太难”，要写“第 7 波平均漏怪 4.2，只剩 18 SAN，边路压力过高”。

推荐指标：

| 指标 | 用途 |
| ---- | ---- |
| 每波剩余 SAN | 判断难度曲线 |
| 每波漏怪数 | 判断防线压力 |
| 每波心魔死亡塔种 | 判断塔是否有存在感 |
| 路线使用占比 | 判断权重分路是否生效 |
| 玩家建塔 / 升级 / 塑形次数 | 判断经济选择是否成立 |
| Boss 谈判选择 | 判断非战斗策略是否有价值 |

当前 CSV 已覆盖前 4 项。玩家建造行为和 Boss 谈判选择仍可作为后续增强项。

### Step 4：配置表外置化（已完成基础闭环）

目标：证明你理解商业项目里的策划配置工作。

当前完成：

- 15 张 CSV 已接入运行时：塔、心魔、波次、出怪、地图、路线规则、路线权重、教学、安全阀、念力残堆、难度曲线和 Boss 战斗参数。
- `docs/config/` 作为策划源表，`public/config/` 作为运行时读取表，`dist/config/` 作为打包产物。
- 启动时读取并校验 CSV；失败时回退 TypeScript 默认值，保证演示不崩。
- 主菜单已有“配置校验”入口，`/?config` 可直接打开报告。

你需要做：

- 用策划视角检查字段是否清楚。
- 给每张表写“为什么需要这些字段”。
- 做一次只改配置不改代码的调参，并记录前后差异。
- 在演示视频里展示配置校验报告和一张关键 CSV。
- 如果要继续加分，把复盘文案模板、Boss 谈判角色和默认对话继续外置。

## 三、简历材料

### Step 5：简历项目描述

建议写法：

> 《认知围城》AI 自适应塔防原型 | 独立游戏策划 / 实现

> - 设计 LLM 复盘 Agent：读取战斗日志，输出下一波策略 JSON，并真实改写路线权重、编队、技能、侵略性和阵容。  
> - 为演示稳定性设计本地 fallback 策略库，使无 API 环境也能完整体验 AI 复盘与 Boss 对话。  
> - 编写 GDD、系统说明和调优计划，沉淀可配置化、数据复盘和多患者内容扩展方案。

如果完成调优记录后，加一条：

> - 接入终局战斗统计 CSV，跟踪每波剩余 SAN、漏怪数、塔种击杀贡献和路线使用占比；后续基于 5 轮试玩数据迭代多路线压力分配。

### Step 6：作品集一页纸

必须包含：

- 项目名、类型、平台、开发周期、个人职责。
- 一张核心截图。
- 一句话玩法。
- 三个设计亮点。
- 一个“问题 -> 方案 -> 结果”的迭代案例。
- 链接：试玩、视频、GitHub、GDD。

三个设计亮点建议：

1. 心魔复盘不是文本装饰，而是下一波真实策略改写。
2. 心魔种类、路线权重和随机扰动共同形成同波分路压力。
3. Boss 谈判把叙事选择转化为战斗参数变化。

## 四、面试讲法

### Step 7：准备 3 分钟项目陈述

结构：

1. 我想解决的问题：传统塔防入侵波次固定，玩家一次找到最优解后重复执行。
2. 我的方案：让心魔方作为 Agent 复盘上一波失败，并真实改写下一波。
3. 我做的系统：战斗日志、复盘策略、EvolutionApplier、路线权重、Boss 谈判、fallback。
4. 我遇到的取舍：全员同路太单调，改成 70/20/10 权重分路。
5. 下一步：配置表、调优数据、演示包和多患者关卡库。

### Step 8：准备高频追问

| 面试官可能问 | 回答重点 |
| ------------ | -------- |
| 为什么用塔防承载 AI？ | 塔防天然有波次、复盘、反制和数值可观测性，适合让 AI 产生可验证影响 |
| LLM 输出不稳定怎么办？ | 有 JSON sanitize、本地 fallback、demoMode，策略字段被限制在可控枚举内 |
| 为什么不是全员按 AI 路线走？ | 早期版本会导致多路线价值被浪费，所以改成策略主权重 + 心魔偏好 + 随机扰动 |
| 如何证明它更好玩？ | 当前要补对照实验：无进化 / 硬路线 / 权重分路，比较漏怪、SAN 和路线压力 |
| 你作为策划做了什么？ | 玩法概念、系统规则、心魔行为、波次节奏、数值框架、GDD、调优计划和演示闭环 |
| 如果给你两周继续做？ | 配置表和数据导出已完成；下一步做 5 轮调参、策略对照实验和稳定演示视频 |
